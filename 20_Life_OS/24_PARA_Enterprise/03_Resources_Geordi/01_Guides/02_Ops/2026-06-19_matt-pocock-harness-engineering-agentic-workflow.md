- **<Harness Engineering — Thèse Centrale>** : Pocock : 'People are focused on the wrong thing. - **<Tactical vs Strategic Programming (Ousterhout)>** : Tactical = écrire du code vite, debug vite, ship vite. Strategic = investir 10-20% de temps upfront pour rendre la code base plus facile à modifier ensuite. Les gens qui dominent avec AI font strategic programming en harness. La différence compoundée sur 12 mois = écart exponentiel de productivité.
- **<Token Economics — Insight Non-Évident>** : Pocock : 'Have a code base that's easier to make changes in.' Harness bien fait = moins de tokens consommés parce que moins de réexploration, moins de bug context. Investir 1 jour dans la doc = économiser 100K tokens sur 6 mois. ROIs du harness engineering sont sous-estimés par les devs qui optimisent uniquement les models.
- **<Matt Pocock Skills Repo = 130K Stars>** : Approche = skills déclaratives + commands structurées, pas prompts ad-hoc. Standard de facto pour harness TypeScript. La communauté a plébiscité parce que c'est le bon niveau d'abstraction — assez structuré pour donner de la cohérence, assez flexible pour s'adapter à chaque projet.
- **<Harness Quality Mesurée par D11 Fable Score>** : L'évaluation du harness doit être mesurable. D11 Fable score = mesure quantitative (coverage + token economy + speed + quality). Sans métrique, le harness dérive vers l'over-engineering ou le sous-engineering. Doctrine A'Space : tout A3 twin doit avoir un score mesurable.
