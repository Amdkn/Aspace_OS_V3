## 🧠 Concepts Clés à Haute Densité Sémantique

- **<Cognitive Load Bottleneck>** : Le plafond de productivité avec les coding agents n'est plus la capacité du modèle, mais l'attention et la charge cognitive de l'hôte qui doit superviser 2-3 sessions interactives en parallèle (worktrees isolés, multi-agents via Super set / Conductor). Le context-switching entre threads provoque des erreurs — envoyer la mauvaise instruction au mauvais agent devient un risque opérationnel quotidien. AI Jason le formule explicitement : "ceiling of how much we can get out from those coding agent is no longer the model capability, but our own attention and cognitive load." C'est le driver paradigmatique qui rend Symphony nécessaire : passer d'une gestion par session à une gestion par livrable.

- **<Ticket-Level Abstraction & State Machine>** : Le pivot conceptuel fondamental de Symphony — déplacer l'humain "up a level". Au lieu de gérer 2-3 sessions interactives, on gère des tickets. Le ticket tracker (Linear, Plane, Jira, Trello) devient la state machine durable qui médiatise toute interaction humain ↔ agent. L'agent travaille au niveau du ticket, rapporte via le ticket lui-même (plan + checklist + video recording), et l'hôte reste in the loop sans monitorer les sessions individuelles. Cette abstraction est "embarrassingly simple but very effective" selon AI Jason — la simplicité conceptuelle cache une réduction massive de friction opérationnelle.

- **<Symphony 3-Component Architecture>** : Le système tient en 3 composants : (1) **scheduler** = background process qui poll le ticket tracker toutes les 30 secondes, setup workspace isolé, manage session lifecycle ; (2) **workflow.md** = fichier version-controlled dans le repo qui contient YAML config (project slug, polling frequency, parallel agent count, hooks programmatiques) + markdown SOP (le prompt agent pour chaque tour — comment planifier la tâche, comment valider le travail, Definition of Done, escalation humaine) ; (3) **external ticket system** (Linear/Plane/etc) = state machine durable pour humain ↔ agent. Pas de UI admin séparée, pas de config service — tout vit dans le workflow.md.

- **<workflow.md Dual Layer Pattern>** : Le fichier workflow.md est l'innovation centrale de Symphony. Sa structure bi-couche combine (a) **YAML front matter** = configuration pure du scheduler (project slug Linear, fréquence poll 30s, max agents parallèles, hooks post-workspace setup, agent settings) ; (b) **markdown body** = Standard Operating Procedure rendu comme prompt agent à chaque tour (comment planifier, comment valider, Definition of Done, conditions d'escalation humaine). Avantages : (1) version-control via PR normaux, (2) pas de drift config/code, (3) onboarding nouvelle capability = éditer un markdown, pas déployer un service, (4) audit trail complet Git. C'est l'inverse du pattern "config file + admin UI + separate orchestrator" classique.

- **<Harness Engineering Prerequisite>** : Symphony ne fonctionne que si le coding agent environment est set up pour compléter des tickets end-to-end atomiquement. AI Jason identifie 3 prérequis : (a) **bootability** — agent peut run un script et tout bootstrap sans y passer du temps ; (b) **documentation structure** — index dans CLAUDE.md / AGENTS.md pointant vers les bonnes ressources ; (c) **self-verifying tools** — agent peut faire un end-to-end test après implémentation ET uploader une preuve (video recording) directement dans le ticket. Le harness engineering n'est pas une feature Symphony — c'est le substrate qui rend Symphony utile. Sans harness, Symphony devient un scheduler de tickets vide.

- **<Self-Verifying Tools & Video Proof>** : L'innovation qui rend la boucle "humain in the loop mais sans monitorer chaque session" vraiment opérationnelle. AI Jason a testé Playwright MCP (trop token-heavy même quand inutile) et trouvé **Playwright CRI tool** comme sweet spot — permet à l'agent de record sa session browser en MP4/WebM, ajouter des chapters annotés via HTML overlay, uploader le résultat dans le ticket Linear comme preuve de travail. Capabilities avancées : annotation des actions agent on-top, chapter rendering, integration Linear upload API. Ni Chrome DevTools MCP ni Agent Browser n'ont cette capability out-of-box. Combiné avec skills spécialisés (Playwright CRI skill, linear skill, server-start skill, Grafana logs skill), l'agent peut capturer et prouver son travail sans intervention humaine pendant l'exécution.

- **<Status-Driven Workflow Lifecycle>** : Le state machine Symphony out-of-box définit 4 statuts Linear spécialisés qui gouvernent le flow : **Todo** = trigger pickup automatique par scheduler (création workspace + démarrage agent session) ; **In Progress** = agent travaille (changement automatique, hôte passif) ; **Human Review** = agent a terminé, hôte review le plan + checklist + video uploadés dans le ticket ; **Merging** = hôte valide, agent crée automatiquement la PR. Cette state machine permet à l'hôte de basculer entre dozens de tickets mentalement sans devoir se souvenir de chaque contexte — il regarde le statut, lit la vidéo si besoin, valide ou rejette. Linear UI peut afficher ça en Kanban (Todo / In Progress / Human Review / Merging), créant une "engineering dashboard" passive.

## 🔬 Annexes Détaillées de l'Audit Sémantique & Technique

### Note A3-01 : Symphony SPEC.md vs Elixir impl — pattern de référence sans clone

Le repo `github.com/openai/symphony` offre 2 entrées : **SPEC.md** (2185 lignes, protocole agnostique) et **elixir/** (reference impl Apache-2.0 en Elixir/Phoenix). AI Jason a testé les deux : (a) demander à un coding agent de lire SPEC.md et d'implémenter en Python (custom build) ; (b) réutiliser directement l'impl Elixir. Conclusion AI Jason : dans 90% des cas, l'impl Elixir suffit. A'Space position : **ne PAS cloner l'Elixir** (stack mismatch — A'Space canon = JS/TS/Supabase/Plane, pas Elixir/Phoenix). Référencer SPEC.md uniquement dans `wiki/hand_offs/handoff_symphony_pilotage_paperclip_anti_maximizer_2026-06-22.md` §1.1 comme pattern architectural. La skill `/symphony-pilot` Phase B sera notre implémentation native en JS/TS sur MCP Plane + Supabase, sans dépendance externe.

### Note A3-02 : Skills ecosystem — mapping vers 35 A3 twins canoniques

AI Jason décrit 5+ skills spécialisés que l'agent charge pour opérations de routine : (a) **Playwright CRI** (video recording browser sessions) ; (b) **linear-skill** (operate Linear API + upload video evidence) ; (c) **server-start** (bootstrapping local server) ; (d) **grafana-logs** (fetch production logs pour bug fixing) ; (e) **debugging-skill** (reusable patterns). A'Space position : ces skills correspondent directement à nos **35 A3 twins canoniques** documentés dans `handoff_a3_data_cartography_v1_2_2026-06-21.md`. Mapping proposé Phase B : (linear skill) ↔ A3 Cerritos Tendi (PARA placement) ; (server-start) ↔ A3 Enterprise Geordi (Resources curation) ; (grafana-logs) ↔ A3 SNW Chapel (Measures) ; (debugging) ↔ A3 Discovery Stamets (H30 mycelium network). Le skill `/symphony-pilot` Phase B dispatchera les A3 twins appropriés selon `--a3-subagent=<name>` arg §3.

### Note A3-03 : 30-second polling vs CronCreate durable — cadence tradeoff

Symphony poll Linear toutes les **30 secondes** pour détecter les tickets Todo. A'Space `/symphony-pilot` heartbeat-interval default = **600s (10 min)** — 20× moins agressif. Pourquoi : (a) **Sobriété** — 30s × 1 agent × 24h = 2880 polls/jour = 1M+ polls/ans, overhead API non négligeable ; (b) **D7 cost-of-escalation** — heartbeat rapide = risque de reaction excessive à des micro-changements de statut (re-create workspace, re-dispatch agent inutilement) ; (c) **D6 lesson #1** — Interval < 60s déclenche throttling CC infrastructure. Le tradeoff : 600s = latence max 10 min entre "ticket Todo créé" et "agent démarre", acceptable pour A0 (pas d'urgence opérationnel immédiate). Phase B pourrait descendre à 60s floor pour les Rocks critiques (12WY W2 sprint). Notion clé : Symphony cadence est configurable par workflow.md YAML ; `/symphony-pilot` cadence est configurable par `--heartbeat-interval=<sec>` arg avec floor 60s.

### Note A3-04 : Auto-PR creation pattern & merge gating — implications A'Space

Symphony crée automatiquement une **PR Git** quand le ticket passe en statut **Merging** (post-Human Review). Cette automation ferme la boucle ticket → code review → production-ready sans intervention humaine. A'Space implications Phase B : (a) Plane a un concept similaire via "modules" mais pas de PR auto-creation native ; (b) le module `d3abd60a-dee5-4be2-99fd-c55cffdcf332` créé 2026-06-22 est l'équivalent A'Space d'un "release container" ; (c) Phase B pourrait ajouter webhook Plane → GitHub PR creation via `mcp__vercel__create_webhook` (events: `deployment.ready`) ou custom Edge Function Supabase. Garde-fou critique : **ADR-SOBER-002 Hard-stop 6** (Musk-style speed > safety) → toute auto-PR doit passer par Human Review status obligatoire, jamais Merging direct. Risk class = ORANGE (high visibility, low blast radius) per `/symphony-pilot` §10 D11 metrics.

### Note A3-05 : Harness Engineering = substrate qui rend Symphony utile — mapping vers A'Space CLAUDE.md/AGENTS.md

AI Jason insiste : Symphony ne fonctionne que si le coding agent environment est set up pour compléter des tickets end-to-end atomiquement. Les 3 prérequis (bootability + documentation structure + self-verifying tools) sont exactement ce que **CLAUDE.md global** + **AGENTS.md canon** + **MEMORY.md index** + **Graphify corpus 203K nodes** font pour A'Space. Le jumelage A0↔A acté 2026-06-21 est la version humaine de ce harness — A (humain) émet les intentions, A0 (jumeau numérique) méta-orchestre, Claude Code A2 infrastructure, A3 sub-agents exécutent.
