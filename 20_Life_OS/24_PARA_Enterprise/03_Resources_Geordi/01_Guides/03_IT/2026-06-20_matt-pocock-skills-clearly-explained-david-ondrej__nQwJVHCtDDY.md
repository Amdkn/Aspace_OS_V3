## 🧠 Concepts Clés à Haute Densité Sémantique

- **Tactical Programming** (John Ousterhout, *A Philosophy of Software Design*) : Démarche quotidienne du développeur focalisée sur l'immédiat — écrire la syntaxe, corriger les bugs au fur et à mesure, faire fonctionner le code. Caractérisée par des décisions locales sans vision d'ensemble, où chaque commit résout un symptôme mais peut introduire une dette technique systémique. **L'AI a complètement eaten tactical programming** (Matt Pocock) — elle l'exécute pour moins cher, plus vite, 24/7, sans fatigue. Conséquence opérationnelle : un développeur cantonné au tactical ajoute moins de valeur qu'un AI agent équivalent. La bataille tactique est perdue d'avance.

- **Strategic Programming** (John Ousterhout, ibidem) : Démarche long-terme focalisée sur l'architecture — comment le codebase doit être structuré pour rester modifiable, quelles interfaces entre modules facilitent l'évolution, quels investissements减速Now paient sur 6-18 mois. Caractérisée par des décisions locales *qui anticipent le changement* (DRY modéré, abstractions时机, naming révélateur d'intention, tests d'intégration aux frontières). **Les seniors avec AI ont 10x boost ; les juniors avec AI ont un boost marginal** — le strategic programming est le ceiling qui détermine si l'AI ajoute 10% ou 1000% de valeur.

- **Harness > Model** : Matt Pocock insiste — *"People are focused on the wrong thing. They're looking at the big shiny new thing when in fact just focus on the stuff that's been working for 30-40 years"*. Le harness = skills + prompts + environment + tooling + codebase structure. Le model = le LLM sous-jacent. L'opérateur a **beaucoup plus de contrôle sur le harness que sur le model**. Optimiser le model choice (GPT-5 vs Claude vs Gemini) est tactique et marginal ; optimiser le harness est stratégique et multiplicatif. Application A'Space : investir dans MCP servers, skills Claude, scripts Hermes, et architecture de l'agentic layer rapporte infiniment plus que de switcher entre providers LLM.

- **Skills as Multiplier** (Matt Pocock, *Total TypeScript*) : *"Skills are a multiplier for AI... if I'm able to oversee a codebase and think about how things should be built and just tell AI how to do it, then AI just has so much richer context to work with."* Une skill = un prompt structuré + outils + sortie attendue, versionné comme du code. La skill teach de Matt génère des curriculums sur n'importe quel sujet en utilisant zone of proximal development, knowledge/skills/wisdom distinction, et resources trusted. Application A'Space : les 49 skills dans `C:\Users\amado\.claude\skills\` sont l'équivalent direct de ce concept — chacune encode une doctrine + workflow + outputs. Le passage de "je sais comment faire" à "la skill sait comment faire" est le multiplicateur strategic.

- **Token Spend Optimization = Codebase Simplicity** : Matt Pocock cite *"how do you optimize for token spend? Have a code base that's easier to make changes in."* L'intuition contre-intuitive : un codebase complexe coûte plus en tokens à chaque itération AI (l'AI doit re-comprendre le contexte, faire des allers-retours, tester plus). Un codebase simple (DRY judicious, naming clair, boundaries nettes) coûte moins en tokens ET produit moins de bugs. **Token economy ≠ minimiser les tokens par requête ; token economy = minimiser le coût total pour atteindre un outcome**. Pour A'Space : la doctrine "small files < 800 lines, immutable updates, no any, Zod validation" (voir `~/.claude/rules/ecc/typescript/coding-style.md`) n'est pas du purisme — c'est du strategic programming qui minimise le coût AI long-terme.

- **Junior Fleet Mental Model** : Matt Pocock décrit l'AI comme *"this infinite fleet of tactical programmers that you now have access to."* Pour en tirer le maximum, il faut être un excellent strategic programmer qui orchestre. Conséquence organisationnelle pour OMK Services : ne plus hire de juniors purs (l'AI les rend obsolètes sur le tactical) mais investir dans des seniors strategic-oriented (LD08 Impact = multiplier élevé par AI fleet). Les cercles de能力 (Cerritos pour capture, Boimler pour clarify, Rutherford pour reflect) sont l'équivalent A'Space de la "junior fleet orchestrating" — chaque agent A3 a un rôle tactique délimité, A0 fournit la vision stratégique.

## 🔬 Annexes Détaillées de l'Audit Sémantique & Technique

### Note A3-01 : La distinction Tactical/Strategic comme grille de lecture pour ADR-META-001 (Doctrine Anti-Paresse)

John Ousterhout distingue tactical programming (décisions locales, court-terme) de strategic programming (investissements long-terme dans la structure du codebase). Matt Pocock applique cette grille à l'AI : "AI has basically eaten tactical programming. It's gone, right? It's all gone. So AI is just better at doing tactical programming than you are because it can do it for cheaper." Conséquence directe pour A'Space : la doctrine ADR-META-001 (Doctrine Anti-Paresse, D1-D8) doit être ré-interprétée. D1 "Verify-Before-Assert" et D4 "No Self-Contradiction" sont des règles strategic — elles préviennent la dette technique cognitive qui rend le codebase hostile aux itérations AI futures. D6 "Persistent Symptom → collect exact command/message" est tactique-avec-discipline — correct pour la réponse immédiate mais ne suffit pas si elle n'est pas suivie d'un strategic fix (exemple de la session 2026-06-20 : les D6 lessons #44 → #60 accumulent 17 fixes tactiques sans qu'un seul ADRs structural émerge). Recommandation : après chaque D6 lesson, A0 doit demander "quel pattern architectural ce D6 lesson révèle-t-il ?" et exiger au moins 1 ADR-META-001 amendment par semaine pour transformer tactical en strategic.

### Note A3-02 : Skills vs Prompts vs Documentation — la triade strategic d'A'Space OS

Matt Pocock : "Skills are a multiplier for AI" et "Skills are more important now than they used to be because again, you just have this multiplier available to you and you can delegate more". A'Space OS a déjà 49 skills dans `C:\Users\amado\.claude\skills\` — c'est un investissement strategic massif. La question stratégique est : **quelle est la hiérarchie doctrine-encoding dans A'Space ?** Niveau 1 (strategic maximum) : CLAUDE.md canonique qui définit D1-D8 + les rôles A1/A2/A3. Niveau 2 (tactical reusable) : les 49 skills qui encodent des workflows spécifiques (youtube-to-guide, ratify-adr, l0-deploy-verify). Niveau 3 (ephemeral) : les prompts in-conversation. Le bug actuel : on observe parfois un mix entre Niveau 2 et Niveau 3 (un prompt ad-hoc qui aurait dû être une skill). Recommandation stratégique : pour chaque session qui produit un workflow > 3 tool calls répété, créer immédiatement une skill (doctrine "Skill Creator Reflex" déjà ancrée). Audit trimestriel : pour chaque skill existante, vérifier qu'elle a été invoquée ≥ 5 fois dans les 90 derniers jours ; sinon, soit la décommissionner (move to `_TRASH`), soit la promouvoir (l'utiliser davantage).

### Note A3-03 : Token Spend Optimization = Codebase Simplicity — la doctrine Rick anti-bloat appliquée à A'Space

Matt Pocock : "how do you optimize for token spend? Have a code base that's easier to make changes in". A'Space OS a un problème récurrent de bloat : 49 skills, 17 sub-agent profiles, 10 MCP servers, 1000+ pages de wiki. Chaque élément est individuellement justifiable mais l'ensemble impose un coût cognitif ET token non négligeable à chaque session. La doctrine Rick "Sobriété & Anti-fragility" (CLAUDE.md ligne "A1 Rick — Architect of Sovereignty (L0 Bedrock/Tech OS). Law: 'Sobriété & Anti-fragility'") exige une audit trimestrielle : pour chaque skill/agent/MCP, calculer le ratio (invocations/30j) / (LOC maintanance) ; si < 1, candidat à la décommission. Application immédiate : la session 2026-06-19 a installé motion@12.23.24 (669 KB) qui n'a jamais été importé ; le 2026-06-20 a décommissionné autoprefixer (390 KB). Ces wins tactiques (1.06 MB libérés) sont stratégiques parce qu'ils réduisent le coût marginal de chaque session AI future. La doctrine : "codebase easier to make changes in" = codebase qui contient moins de surface à maintenir.

### Note A3-04 : "Senior 10x vs Junior 1.1x" appliqué à l'architecture agentique A'Space

Matt Pocock : "AI makes senior developers just 10 times better. And it sort of doesn't make sense to hire that many juniors anymore because juniors get a little boost from AI, but seniors just get this ridiculous huge boost from it and they can do so much more with it. So your skills are the ceiling on what AI can do." Traduction A'Space : les A3 twins (35 profils) sont les "juniors" tactiques — chacun a un scope délimité (Boimler = clarify, Freeman = engage, Rutherford = reflect). Les A2 (6 ships : Curie, Zora, LCARS Computer, Holo Deck, Holo Janeway + Picard en réalité A3 Projects) sont les "seniors" qui orchestrent plusieurs A3. Les A1 (3 ships : Beth, Morty, Rick) sont les "CTO/strategic operators". L'architecture est correcte (junior fleet orchestrée par seniors) mais souffre d'un déséquilibre : **les A3 sont sur-employés en tactical, les A2 sont sous-employés en strategic orchestration**. Recommandation : pour chaque A2 ship, définir 1-2 "capabilities strategic" qu'il/elle orchestre activement (ex : ZORA = balance checker across LDxx ; Curie = execution cadence across sprints). Mesurer le ratio strategic/tactical par A2 chaque semaine.

### Note A3-05 : Le /teach skill de Matt Pocock comme archétype de skill strategic A'Space

Matt Pocock décrit le teach skill : "I take some of the teaching principles that I know about such as the zone of proximal development, such as the difference between knowledge skills and wisdom, encode that into a skill and essentially use it to create a course on the fly about any topic". Ce skill est l'archétype strategic : il encode de la doctrine pédagogique (Lev Vygotsky, Bloom's taxonomy) + workflow itératif + outputs structurés (curriculum). Application A'Space : les skills existantes devraient être évaluées selon ce critère — **encodent-elles de la doctrine + workflow + outputs structurés, ou sont-elles du tactical one-off déguisé ?** Le teach skill réussit parce qu'il encode 50+ ans de recherche pédagogique (Vygotsky, Bloom, Knowles). Les skills A'Space qui encodent 5+ ans de doctrine (ex : ratify-adr encode ADR-META-001 D1-D8, youtube-to-guide encode Antigravity Premium Standard) sont strategic. Les skills qui encodent juste "comment faire X avec Y outils" sans doctrine sous-jacente sont tactical déguisé. Recommandation : audit stratégique — pour chaque skill, écrire 1 paragraphe "doctrine encodée" ; si la skill ne peut pas articuler ce paragraphe, la candidate à la décommission ou la promotion (doctrine d'abord, skill ensuite).
