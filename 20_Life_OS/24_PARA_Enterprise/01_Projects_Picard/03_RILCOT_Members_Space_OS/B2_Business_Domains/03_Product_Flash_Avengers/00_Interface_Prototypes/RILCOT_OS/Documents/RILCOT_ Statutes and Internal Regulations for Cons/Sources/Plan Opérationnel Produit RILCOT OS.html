Magistral, Capitaine. Tes maquettes Stitch posent déjà l’ossature du RILCOT OS : accueil public, dashboard membre, gouvernance, portefeuille projets, réseau universitaire, finances. Ça colle proprement au SOB que nous avons formalisé.

Voici un plan d’attaque ultra-opérationnel pour passer du proto au produit :

1) Architecture produit (vue d’ensemble)

  • Apps :

  1. Public (site RILCOT) → Mission, Structure, Domaines, Adhésion, Contact.

  2. Membres (Dashboard) → Adhésion (Demande/Paiement/Statut), Profil, Agenda, Documents.

  3. Admin/CC → Gouvernance (AG/CC/CF), Projets, Réseau universitaire, Finances/KPI.

  • Rôles/RBAC : Public, Membre, Secrétaire Exécutif, Trésorier, Coordonnateur, CF. (Audit log + consentements).

2) Parcours critiques (alignés SOP)

  • SOP-01 – Adhésion (tes onglets Demande / Paiement / Statut sont parfaits) :
  1. Saisie → 2) Paiement → 3) Validation CC → 4) ID + Carte → 5) Onboarding → 6) Cotisation mensuelle.

UX à ajouter : barre de progression, validations live, récap PDF, reçus, relances auto J+3/J+30.

  • SOP-02 – AG : calendrier, quorum, PV signé, publication décisions (liste consultable + filtres).

  • SOP-03 – Dépenses : double signature, pièces jointes, rapprochement, variance budget. (Widgets “exécution budgétaire” déjà dans ta planche).

  • SOP-05 – Portefeuille projets : cartes “idée→cadrage→exécution→revue→clôture”, KPI impact et ODD visibles.

  • SOP-08 – Réseau universitaire : fiches points focaux, feuilles de route trimestrielles, kits d’essaimage téléchargeables.

3) Modèle de données minimal (DB)

  • Members(id, nom, email, tel, campus, statut, roles)

  • Applications(id, member_id, formulaire_json, statut, date)

  • Payments(id, member_id, type: adhesion|cotisation, montant, devise, reçu_url, date)

  • Projects(id, titre, phase, budget, depense, kpi_cle, responsable_id)

  • Meetings(id, type: AG|CC|CF|CR, date, quorum, pv_url, decisions[])

  • Decisions(id, meeting_id, objet, vote, resultat, lien_preuve)

  • Universities(id, nom, ville, pays) ; Representatives(member_id, university_id, mandat_debut/fin)

  • Documents(id, type, url, access_level)

  • AuditLogs(id, actor_id, action, entity, timestamp)

4) API/Endpoints (extrait)

  • POST /applications (création adhésion) → GET /applications/:id/status

  • POST /payments (webhook PSP) ; GET /members/:id/payments

  • GET /projects?phase=… ; POST /meetings ; POST /decisions

  • GET /kpi/summary?period=YYYY-MM (alimente les cartes KPI).

5) Composants UI (librairie)

  • Atoms : Badge Statut (Actif/Suspendu/En révision), Tag Domaine, ProgressBar % budget.

  • Molecules : Card Projet (phase + %), Item Finance (montant + double signature OK/NOK).

  • Organisms : Wizard Adhésion (3 étapes), Tableau Décisions (filtre AG/CC/CF), Feuille de Route Campus.

  • Templates : Dashboard Membre (header + tabs), Console Admin (nav latérale + sections Gouvernance/Projets/Finances).

Ces blocs recouvrent précisément tes écrans de “Gouvernance / Portefeuille / Réseau universitaire / Finance Overview”.

6) Mesure & KPI (widgets à implémenter)

  • Gouvernance : AG tenues (4/an), quorum (%), délai PV (jours).

  • Communauté : membres actifs, rétention, ratio genre/jeunes.

  • Projets : livrés/phase, % livrables atteints, bénéficiaires.

  • Finances : exécution budgétaire %, fonds levés, conformité double signature.

  • Campus : # actifs, événements/trimestre, adhésions étudiantes.

7) Sécurité & conformité

  • Double signature en workflow (décision + paiement ≠ même personne).

  • Journalisation (AuditLogs) par action sensible.

  • Charte veto : pop-in informative + log “motif + durée + réversibilité” (revu par CF).

8) Qualité d’exécution (checklist design/tech)

  • Accessibilité (contrastes AA, focus, labels).

  • Mobile-first (tes écrans y sont déjà, garde les boutons d’action “thumb reachable”).

  • I18N (fr/en) dès le design system.

  • États vides / erreurs / skeleton loaders.

  • Pièces jointes : prévisualisation + vérif taille/type.

9) Backlog “Semaine 0 → Semaine 4”

  1. Découper tes écrans Stitch en composants (atoms→organisms).

  2. Implémenter le Wizard d’Adhésion (3 étapes) + reçu PDF.

  3. Paiement : intégrer un PSP local (Mobile Money) + webhook POST /payments.

  4. Ecran Admin > Gouvernance : créer/répertorier AG/CC/CF, charger PV, créer décisions.

  5. Portefeuille Projets : CRUD + phases + KPI clé.

  6. Finance Overview : budget vs dépense + alertes variance.

  7. Réseau Universitaire : annuaire + feuille de route trimestrielle + export.

  8. KPI Dashboard : brancher 6–8 indicateurs phares.

  9. RBAC + Audit log.

  10. Tests e2e des SOP-01/02/03/05.

10) Copy & micro-interactions (quick wins)

  • Accueil : bouton primaire “Devenir Membre”, secondaire “Découvrir nos projets”.

  • Statuts de dossier : “Reçu → En revue CC → Accepté → Carte prête → Actif”.

  • Finance : badge “Double signature validée” / “En attente 2/2”.

  • Réseau Universitaire : CTA “Télécharger le Kit d’essaimage”.
